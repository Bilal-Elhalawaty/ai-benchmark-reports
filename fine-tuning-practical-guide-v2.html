Fine-Tuning in Plain Language

The Book Analogy: A base AI model is like a brilliant new employee who has read every book ever written. Fine-tuning is giving that employee a custom training manual specific to your company — your database tables, your column names, your business rules. After reading 500 examples of "when a user asks X, the correct SQL is Y," the model learns your patterns.

Fine-tuning does NOT change the model's core intelligence. It teaches the model your specific patterns so it consistently generates correct SQL for your database schema, in both English and Arabic.

Three Methods: SFT, LoRA, QLoRA

These are the three approaches to fine-tuning. You don't always get to choose — provider-hosted services pick for you.

MethodWhat It DoesCostWho Uses It
SFT
Supervised Fine-Tuning		 Retrains the entire model on your examples. Like rewriting the whole textbook. Most expensive Google (Gemini), OpenAI (GPT) — they handle everything, you just upload data
LoRA
Low-Rank Adaptation		 Adds a small "adapter" layer (1-5% of model). Like adding sticky notes to the textbook instead of rewriting it. Medium Together AI (managed), Hugging Face AutoTrain
QLoRA
Quantized LoRA		 Same as LoRA but compresses the model to use 4x less memory during training. Same quality output. Cheapest Self-hosted on rented GPUs (Qwen 3, MiniMax)
Key point: SFT, LoRA, and QLoRA are techniques, not software packages. They are methods for how a model gets trained. The actual tools that implement these methods are things like Vertex AI (Google), OpenAI API, Unsloth, Hugging Face AutoTrain, etc.

Training Server vs Inference Server

This is a common source of confusion. Training and inference are two completely different operations.

Training Server

Purpose: Teach the model your data

Runs: Once (or when you update training data)

GPU needed: Heavy (80GB+ VRAM)

Duration: 30 min – 24 hours

Output: A small adapter file (200–500 MB)

Cost model: Pay per hour of training

Inference Server

Purpose: Answer user questions (production)

Runs: 24/7 while your app is live

GPU needed: Moderate (can use smaller GPU)

Duration: Always running

Input: Loads base model + your adapter file

Cost model: Pay per month (or per token via API)

They are separate: You can train on Hugging Face AutoTrain, then download the adapter and deploy it on a completely different server (RunPod, Lambda, your own hardware). Training and serving don't need to be on the same machine.

The Two Worlds of Fine-Tuning

Provider-Hosted

Google or OpenAI trains the model for you

You upload a file and click "Train"

No GPU to manage, no code to write

Model stays on their servers

Best for: Gemini 2.5 Pro, GPT 4.1

Self-Hosted

You train and run the model yourself

Need a GPU server (rent or own)

Or use Hugging Face AutoTrain (browser-based)

You own the model — no vendor lock-in

Best for: Qwen 3 32B, MiniMax M2.5

Provider-Hosted Fine-Tuning

With provider-hosted, you never touch a GPU. You prepare a training data file, upload it to the provider, click train, and get a new model endpoint. Your existing code just changes the model name.

Gemini 2.5 Pro — Google Vertex AI

Recommended for production. Highest accuracy ceiling. Score: 88/100 base, expected 93–97 after fine-tuning.

Step-by-Step: How to Fine-Tune Gemini

1

Prepare Your Training Data

Create a JSONL file with your question-SQL pairs. Each line is one training example in Gemini's format (uses "model" instead of "assistant", and wraps content in "parts"). See the Training Data tab for the exact format.

2

Upload to Google Cloud Storage

Where: Go to console.cloud.google.com → Cloud Storage → Create a bucket (or use existing) → Upload your .jsonl file.

Your file will have a path like: gs://your-bucket/training_data.jsonl

3

Start Training

Where: Go to console.cloud.google.com → Vertex AI → Generative AI Studio → Tuning.

Click "Create tuning job", select gemini-2.5-pro as base model, point to your GCS file, set epochs to 3, and click "Start Tuning".

Alternative: You can also start training with ~10 lines of Python using the Vertex AI SDK, but the Console UI does the exact same thing.

4

Wait 30–60 Minutes

Google trains the model on their infrastructure. You can monitor progress in the Vertex AI Console. No GPU needed on your side.

5

Use Your Fine-Tuned Model

When done, you get a tuned model endpoint. In your code, change the model name from gemini-2.5-pro to your new endpoint ID. Everything else stays the same.

Self-Hosted Fine-Tuning

Self-hosted means you train the model yourself instead of relying on Google or OpenAI. This applies to open-source models like Qwen 3 32B and MiniMax M2.5 where the model weights are freely available.

Why self-host? Zero per-token cost after setup, full data privacy (training data never leaves your server), no vendor lock-in, and ability to customize everything. The trade-off: you manage the infrastructure.

Three Paths for Self-Hosted Training

Path A — Easiest
Hugging Face AutoTrain

Browser-based. No code. Upload your data file, pick settings, click Train.

Best for: Qwen 3 32B

Path B — More Control
Rented GPU Server

Rent a GPU on RunPod/Lambda. Run a Python training script (~50 lines). Full control.

Best for: Qwen 3 32B, MiniMax M2.5

Path C — Managed LoRA
Together AI

Middle ground: LoRA fine-tuning without managing GPUs. Upload file + API call.

Best for: Qwen 3 32B only

Path A: Hugging Face AutoTrain (No Code)

Best option if you want self-hosted training without writing code. Everything happens in your browser on Hugging Face's website. They provide the GPU, you just upload data and configure settings.

Step-by-Step

1

Go to Hugging Face AutoTrain

Where: huggingface.co/autotrain → Sign in → Click "Create New Project"

2

Configure Your Training Job

Select "LLM SFT" as the task. Choose Qwen/Qwen3-32B as base model. Upload your training JSONL file (OpenAI format works directly). Select QLoRA as the training method. Pick your GPU (A100 recommended for 32B model).

3

Click "Start Training"

Hugging Face provisions a GPU server, loads the model, and trains it with your data. You can watch logs in real time. Takes 4–8 hours for Qwen 32B with 500 examples.

4

Download Your Adapter

When done, the trained LoRA adapter (~200–500 MB) is saved to your Hugging Face account. You can download it and deploy it on any server running vLLM or Ollama.

Path B: Rented GPU (RunPod / Lambda Labs)

For developers who want full control. You rent a GPU server by the hour, SSH in, and run a training script. This is the only option for MiniMax M2.5 and gives maximum flexibility for all models.

Step-by-Step

1

Rent a GPU Server

Where: Go to runpod.io or lambdalabs.com → Sign up → Choose a GPU pod:

• For Qwen 3 32B: 1x A100 80GB (~$1.50/hr on RunPod)

• For MiniMax M2.5: 4x A100 80GB (~$6.00/hr on RunPod)

Click "Deploy" and wait ~2 minutes for the server to start. You get SSH access and a Jupyter notebook.

2

Install Training Tools

SSH into your server and install the required Python packages: unsloth (for efficient QLoRA training), transformers, peft, and trl (all from Hugging Face). This is a single pip install command.

3

Upload Data & Run Training Script

Upload your JSONL training file to the server. Run a ~50 line Python script that loads the base model in QLoRA 4-bit mode and trains it on your data. The script uses the unsloth library which handles all the optimization automatically.

4

Save & Deploy

After training, save the merged model. Deploy it with vLLM — this creates an OpenAI-compatible API server. Your existing code that calls OpenAI or Gemini can be pointed to this server by just changing the base URL and model name.

Path C: Together AI (Managed LoRA)

Middle ground: You get LoRA fine-tuning (cheaper than SFT) without managing GPUs. Upload your file via API, Together AI handles the rest. Only supports select models including Qwen 3 32B.

How It Works

Upload your training JSONL file (OpenAI format) via Together AI's API or dashboard. Select the base model and click "Start Training." Together AI runs LoRA fine-tuning on their infrastructure. When done, call the fine-tuned model via Together AI's API.

Training Data Format

Write once, use everywhere. Write your training data in OpenAI format — it works directly for GPT 4.1, Qwen 3 32B, MiniMax M2.5, Together AI, and Hugging Face AutoTrain. Only Gemini needs a simple conversion (different JSON structure).

The Two Formats

OpenAI Format (Universal)

Works for: GPT 4.1, Qwen 3, MiniMax M2.5, Together AI, HF AutoTrain

training_data.jsonlOne line per example
{"messages": [
  {"role": "system", "content": "You are an SQL assistant..."},
  {"role": "user", "content": "How many active employees?"},
  {"role": "assistant", "content": "```sql\nSELECT COUNT(*)...\n```"}
]}
Gemini Format

Works for: Gemini 2.5 Pro (Vertex AI) only

training_data_gemini.jsonlOne line per example
{"systemInstruction": {"parts": [{"text": "You are an SQL assistant..."}]},
 "contents": [
  {"role": "user", "parts": [{"text": "How many active employees?"}]},
  {"role": "model", "parts": [{"text": "```sql\nSELECT COUNT(*)...\n```"}]}
]}

Key Differences

AspectOpenAI FormatGemini Format
System prompt{"role": "system"} in messages arraySeparate "systemInstruction" field
AI role name"assistant""model"
Content wrappingDirect: "content": "text"Wrapped: "parts": [{"text": "..."}]
Used byGPT, Qwen, MiniMax, Together AI, AutoTrainGemini (Vertex AI) only

Practical Workflow

1. Write all your training examples in OpenAI JSONL format (one file)

2. Use that file directly for GPT 4.1, Qwen 3 32B, MiniMax M2.5, and Together AI

3. Run a simple conversion script to create the Gemini version (same content, different JSON structure)

Result: 2 files from 1 source — same training data, all models covered

The conversion from OpenAI to Gemini format is a straightforward JSON restructuring. A ~20-line Python script handles it automatically. The actual training content (schema, questions, SQL answers) is identical across all formats.
